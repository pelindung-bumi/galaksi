# galaksi

Galaksi is the GitOps repository for the Pelindung Bumi Kubernetes platform.

## What this repo manages

- Argo CD bootstrap and self-management
- foundation platform applications
- environment-driven ApplicationSet workloads
- shared base values and environment overlays

Current production environment:

- project: `pelindung-bumi-prod`
- environment path: `overlays/pelindung-bumi/prod`

## Repository layout

```text
argocd/
  app/        # foundation apps created early
  appsets/    # generic ApplicationSet definitions
  bootstrap/  # day-0 bootstrap and destroy scripts
  projects/   # AppProject manifests
  root/       # root bootstrap application definitions

base/         # shared values/manifests across environments
overlays/     # environment-specific values/manifests
tmp/          # temporary local test manifests
```

## Development shell

This repo uses Nix flakes with `devenv` for the local development shell.

Common tools are available from the flake, including:

- `kubectl`
- `argocd`
- `argonaut`

Enter the shell from the repo root with:

```bash
nix develop --no-pure-eval
```

If you use `direnv`, you can also allow the environment with:

```bash
direnv allow
```

After entering the shell, you can verify the tools with:

```bash
kubectl version --client
argocd version --client
argonaut --help
```

## Bootstrap flow

Bootstrap is designed to start from an empty cluster.

`argocd/bootstrap/bootstrap.sh` does this:

1. create namespace `argo`
2. install Argo CD from Helm chart `9.4.15`
3. wait for Argo CD CRDs and API readiness
4. apply repo credentials
5. apply bootstrap app `root`

Then Argo CD continues the chain:

1. `root`
2. `init`
3. `argocd/projects`
4. `argocd/app`
5. `argocd/appsets`

## Foundation apps

Static apps in `argocd/app/` are created before dynamic appset workloads.

Current foundation apps:

- `argo-system`
- `namespace`
- `gateway-api`
- `cert-manager`
- `rook-ceph-operator`
- `rook-ceph-cluster`

## Dynamic apps

Dynamic apps are generated by `argocd/appsets/pelindung-bumi.yaml`.

Discovery pattern:

- `overlays/pelindung-bumi/*/*/app.yaml`

Minimal `app.yaml` shape:

```yaml
application: envoy-gateway
destinationNamespace: envoy-gateway-system
createNamespace: false
sources:
  - repoURL: https://github.com/pelindung-bumi/samudera
    targetRevision: envoy-gateway-1.7.1
    path: charts/envoy-gateway
    helm:
      valueFiles:
        - $values/base/envoy-gateway/values/values-common.yaml
        - $values/overlays/pelindung-bumi/prod/envoy-gateway/values/values.yaml
  - repoURL: https://github.com/pelindung-bumi/galaksi
    targetRevision: HEAD
    ref: values
```

Optional per-app sync options are also supported:

```yaml
syncOptions:
  - ServerSideApply=true
```

## Base and overlay convention

Use `base/` for reusable defaults.

Use `overlays/` for cluster or environment specific settings.

Examples:

- `base/rook/operator/values/values-common.yaml`
- `overlays/pelindung-bumi/prod/rook/operator/values/values.yaml`
- `base/namespace/manifests/namespaces.yaml`
- `overlays/pelindung-bumi/prod/namespace/patch-namespace.yaml`

## Important current behavior

- `gateway-api` uses the `experimental` CRDs from `samudera`
- `envoy-gateway` is appset-managed
- `envoy-gateway` supports per-app `syncOptions` and currently uses `ServerSideApply=true`
- `rook-ceph-operator` uses `ServerSideApply=true` because its CRDs are large
- `gateway-api` uses `ServerSideApply=true` because its CRDs are large
- Argo CD health is customized so `CephCluster` is treated as healthy when it is `Ready` and `Created`

## Rook Ceph notes

This repo is currently configured for a single-node cluster.

Important details:

- `CephBlockPool` replica size is `1`
- `rook-ceph-block` is the default `StorageClass`
- the node must expose kernel modules through `/lib/modules/<kernel>` for CSI RBD
- on NixOS `kube01`, this is handled from the `semesta` repo, not from this repo

## Bootstrap and cleanup commands

Bootstrap:

```bash
./argocd/bootstrap/bootstrap.sh
```

Destroy everything managed by this stack:

```bash
./argocd/bootstrap/destroy.sh
```

The destroy script is aggressive and intended for full cleanup and rebuild workflows.
