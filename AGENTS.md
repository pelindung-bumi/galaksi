# AGENTS

This file explains how AI agents should work in this repository.

## Purpose

This repository manages the Pelindung Bumi Kubernetes platform through Argo CD.

Agents should preserve the current design:

- day-0 bootstrap via `argocd/bootstrap/`
- one root bootstrap chain: `root` -> `init`
- foundation apps in `argocd/app/`
- dynamic environment apps in `argocd/appsets/`
- reusable defaults in `base/`
- environment-specific settings in `overlays/`

## Current architecture

Bootstrap chain:

1. `argocd/bootstrap/bootstrap.sh`
2. `argocd/bootstrap/app.yaml` creates `root`
3. `argocd/root/init.yaml` creates `init`
4. `init` manages:
   - `argocd/projects`
   - `argocd/app`
   - `argocd/appsets`

Foundation apps in `argocd/app/`:

- `argo-system`
- `namespace`
- `gateway-api`
- `cert-manager`
- `rook-ceph-operator`
- `rook-ceph-cluster`

Dynamic apps:

- generated by `argocd/appsets/pelindung-bumi.yaml`
- discovered from `overlays/pelindung-bumi/*/*/app.yaml`

## Repo conventions

### 1. Keep foundation apps out of the ApplicationSet

If something must exist first, keep it as a plain `Application` in `argocd/app/`.

Examples:

- namespace creation
- gateway-api
- cert-manager
- rook-ceph

Do not move these into the dynamic appset unless the deployment model changes deliberately.

### 2. Keep the ApplicationSet generic

Do not hardcode app-specific branching inside `argocd/appsets/pelindung-bumi.yaml`.

Allowed dynamic fields in overlay `app.yaml`:

- `application`
- `destinationNamespace`
- `createNamespace`
- `sources`
- optional `syncOptions`

Do not reintroduce hardcoded `if eq .application ...` logic unless truly unavoidable.

### 3. Prefer base + overlay split

Use `base/` for reusable values and manifests.

Use `overlays/` for cluster-specific or environment-specific differences.

Examples:

- `base/rook/cluster/values/values-common.yaml`
- `overlays/pelindung-bumi/prod/rook/cluster/values/values.yaml`

### 4. Namespace management

Namespaces are intentionally managed as a foundation app:

- `argocd/app/namespace.yaml`

Namespace manifests live here:

- `base/namespace/manifests/`
- `overlays/pelindung-bumi/prod/namespace/`

Use overlay patch files like `patch-namespace.yaml` for labels such as:

- `environment`
- `cluster`

### 5. Server-side apply

Use `ServerSideApply=true` only where needed, mainly for large CRDs.

Already needed here:

- `argocd/app/gateway-api.yaml`
- `argocd/app/rook-ceph-operator.yaml`
- `overlays/pelindung-bumi/prod/envoy-gateway/app.yaml`

Do not enable SSA globally without a reason.

### 6. Rook/Ceph specifics

This cluster is currently single-node.

Important facts:

- `rook-ceph-cluster` and `rook-ceph-operator` stay separate applications
- `CephBlockPool` is intentionally single replica for now
- Argo CD health is customized so `CephCluster` is healthy when it is `Ready` and `Created`

Host-level RBD support is handled in the `semesta` repo, not here.

### 7. External repos

This repo depends on:

- `samudera` for vendored charts and CRDs
- `semesta` for node OS configuration

When changing chart paths, versions, or host assumptions, check those repos too.

## Working rules for future agents

- Do not reintroduce a special bootstrap `AppProject` unless explicitly requested
- Do not rely on undocumented assumptions about Argo CD readiness; check real cluster state
- Do not convert foundation apps into ApplicationSet items unless explicitly requested
- Do not update `structure.md` unless the user asks for it
- Prefer small, explicit repo changes over abstract refactors
- If destroy/bootstrap behavior changes, update `README.md` and this file
